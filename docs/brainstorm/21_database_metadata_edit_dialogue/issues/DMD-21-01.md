# DMD-21-01 — Reuse the Assign Channels panel outside the route

Type: AFK

User stories covered: 2, 3, 4

## What to build

Extract the existing Assign Channels tab behavior from the `/database/edit` route into a reusable scope-driven panel that can be embedded by both the full-page route and the metadata edit dialog. The full-page route should continue to behave as it does today, including loading channel-map editor data, rendering the plot mapping table and CSV preview, validating numeric column inputs, saving through the channel-map API, showing missing-channel-map messaging, and enforcing write permissions.

The slice is complete when the current route tab is backed by the reusable panel and there is no user-visible behavior regression.

## Acceptance criteria

- [ ] The full-page Assign Channels tab still loads and saves channel maps for a selected program/version.
- [ ] The reusable panel accepts an externally supplied program/version scope without depending on the route side panel.
- [ ] The current route can still own its side-panel selection state and pass the selected scope into the panel.
- [ ] Existing load, save, disabled-save, toast, and missing-channel-map footer behavior is preserved.
- [ ] Users without write access cannot save channel maps from the panel.
- [ ] Existing channel-map save query invalidation still runs after a successful save.
- [ ] The panel exposes an `onDirtyChange` callback (or equivalent) so a parent dialog can track unsaved channel-map edits.
- [ ] Focused tests cover the extracted panel or route wrapper enough to catch regressions in scope initialization, save enablement, and dirty reporting.

## Blocked by
