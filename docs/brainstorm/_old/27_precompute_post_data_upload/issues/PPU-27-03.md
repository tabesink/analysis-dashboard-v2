# PPU-27-03 — Rescale scheduled damage for schedule-only edits

Type: AFK

Blocked by: PPU-27-02

User stories covered: 3, 12, 13, 14, 15, 17, 18, 29, 30, 35, 40

## What to build

Add a rescale-only path for durability schedule edits that change only schedule scaling inputs. When persisted base damage is current for the relevant event/channel rows, update scheduled damage from the notebook-defined formula without rerunning py-fatigue:

```text
scheduled_damage = base_damage * repeats * weight * multiplier
```

The rescale service should be isolated behind a small interface and used by the precompute orchestrator. It must never treat stale or error base damage as current, and it must fall back to full damage calculation when event matching, raw load histories, channel assignments, channel metadata, or other base-damage inputs changed.

## Acceptance criteria

- [x] Schedule-only edits to repeats, weight, or multiplier can update persisted scheduled damage without starting py-fatigue.
- [x] Rescale-only updates scheduled damage using `base_damage * repeats * weight * multiplier`.
- [x] Rescale-only updates row metadata needed to keep latest damage state current, including schedule identity, status, stale reason, and updated timestamp.
- [x] Rescale-only is allowed only when current base damage exists for every relevant event/channel row.
- [x] Stale or error base damage rows are never used for rescale-only updates.
- [x] Schedule changes that affect event matching fall back to full `damage_calculation`.
- [x] Channel mapping, raw load history, channel metadata, or base-damage input changes fall back to full `damage_calculation`.
- [x] Existing stale successful damage remains inspectable if rescale-only is blocked or fails.
- [x] Server tests cover eligible rescale, missing base damage, stale base damage, error base damage, event-match changes, and formula correctness.
