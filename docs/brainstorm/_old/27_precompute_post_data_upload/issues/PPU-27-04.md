# PPU-27-04 — Backfill missing damage from Inspect Damage for write users

Type: AFK

Blocked by: PPU-27-01

User stories covered: 4, 5, 6, 10, 11, 19, 20, 21, 24, 27, 31, 38, 39, 40

## What to build

Teach Inspect Damage to treat missing persisted damage with ready prerequisites as repairable state for write users. Inspect Damage should continue reading persisted rows only; it must not compute damage synchronously while building the table. For authorized write users, the page may start or reuse the existing `damage_calculation` task through the precompute orchestrator and show progress using the existing derived-data task UX. Read-only users should never start mutation work and should only see the persisted state they are authorized to view.

## Acceptance criteria

- [x] Inspect Damage continues to read persisted `event_channel_damage` rows and does not compute fatigue synchronously on read.
- [x] A write user opening Inspect Damage with missing persisted damage and ready prerequisites starts or reuses a `damage_calculation` task.
- [x] A write user opening Inspect Damage with blocked prerequisites does not start a task and receives toast-only blocked feedback.
- [x] A read-only user opening Inspect Damage never starts a background write or mutation task.
- [x] Active-task reuse prevents repeated page visits from launching duplicate damage tasks.
- [x] Long-running automatic backfill uses the existing derived-data modal behavior, including close-and-continue processing.
- [x] Task ownership and polling constraints remain unchanged.
- [x] Client and server tests cover write-user backfill, read-only no-op, active-task reuse, blocked prerequisites, and no synchronous compute-on-read.
