# UP-24-06 — Harden derived-data task flows and document rollout

Type: AFK

Blocked by: UP-24-02, UP-24-04, UP-24-05

User stories covered: 8, 9, 10, 11, 32, 33, 34, 35, 40, 41, 42, 43, 44, 45

## What to build

Harden the completed channel reprocess, schedule damage, and Inspect Damage flows as one lean upload-pipeline feature. This slice should close cross-flow gaps, document the behavior, and verify the app does not grow into a heavier job system.

Focus on end-to-end consistency: active-task reuse, creator-scoped polling, close-only modal behavior, stale result behavior after channel/schedule changes, failure-report repair loops, timeout-safe UX, and documentation for future agents.

## Acceptance criteria

- [ ] End-to-end verification covers upload without channel map, Assign Channels reprocess, schedule upload/save, damage calculation, stale marking, repair report, retry, and Inspect Damage display.
- [ ] Long-running channel reprocess and damage calculation do not depend on a single long synchronous browser request.
- [ ] Closing a progress modal consistently leaves processing running and discoverable through the scoped inline banner.
- [ ] Active-task reuse is consistent across channel reprocess and damage calculation.
- [ ] Creator-scoped polling is enforced for derived-data tasks.
- [ ] Existing folder upload behavior is not regressed by the upload-task storage extension.
- [ ] Existing channel-map LTTB output remains unchanged apart from async progress behavior.
- [ ] Existing durability schedule upload/save behavior remains intact while adding damage task integration.
- [ ] Existing Inspect Damage base damage calculation remains numerically consistent when persisted through the new path.
- [ ] Documentation explains that upload/canonicalization owns raw load histories, channel assignment owns cross-plot data, and schedule assignment owns load-history damage.
- [ ] Documentation states that failure reports are transient and `event_channel_damage` is a latest-result cache, not an audit ledger.
- [ ] `CHANGELOG.md`, task notes, and build-plan tracking are updated if this work is adopted into the tracked roadmap.
- [ ] A decision-log entry is added only if implementation introduces a durable architectural decision beyond the PRD.
