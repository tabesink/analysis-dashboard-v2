# UPF-29-02 — Harden long-running derived-data polling

Type: AFK

Blocked by: None

User stories covered: Architecture review finding H2

## What to build

Make channel reprocess and damage calculation polling resilient enough for large local-network jobs. Derived-data task polling should tolerate transient server or network interruptions for the same practical window as folder upload polling, and it should communicate retry state without falsely presenting a still-running task as failed.

This slice should not change the server task model. It should improve the client polling contract and tests around retry/backoff behavior.

## Acceptance criteria

- [ ] Derived-data task polling uses a retry window appropriate for long-running reprocess/damage jobs.
- [ ] Transient 5xx, 408, 429, and network errors keep retrying with bounded backoff while the retry window remains open.
- [ ] A terminal `completed` or `failed` task status still resolves immediately once observed.
- [ ] Non-retryable errors still surface promptly with actionable copy.
- [ ] The progress modal or stored progress message indicates temporary connection loss while polling is retrying.
- [ ] Tests cover transient interruption recovery and retry-window expiry.
- [ ] Folder upload polling behavior is not regressed.

## Blocked by

None - can start immediately
