# DMD-20-03 — Harden dialog save, permissions, dirty-close, and refresh behavior

Type: AFK

Blocked by: DMD-20-02

User stories covered: 3, 4, 5

## What to build

Make the metadata edit dialog production-ready by aligning its save path, permissions, dirty-close handling, accessibility, and refresh behavior with the existing Edit Metadata workflow. The dialog should be safe to use for real metadata edits from the Database table and should not silently discard changes or leave stale metadata visible after save.

## Acceptance criteria

- [ ] Saving from the dialog uses the existing program/version metadata update contract.
- [ ] Successful saves invalidate all metadata-dependent client queries used by the Database table, Dashboard filters, and Edit Metadata workflow.
- [ ] The Database table reflects saved metadata changes without requiring a manual page refresh.
- [ ] Admin-only fields such as Status follow the same rules as the full-page Edit Metadata route.
- [ ] Users without write permission cannot save metadata from the dialog and receive the same style of disabled affordance or explanation used elsewhere in the app.
- [ ] Closing the dialog with dirty metadata values follows the app's established unsaved-change behavior; dirty changes are not silently lost.
- [ ] Focus is trapped in the dialog while open, Escape/close behavior is predictable, and the initial focus target is sensible.
- [ ] Keyboard-only users can open the pencil action, navigate the left rail, edit fields, save, and close.
- [ ] Tests cover successful save invalidation, permission-gated fields/actions, dirty-close behavior, and accessible dialog controls.
- [ ] Manual verification covers opening multiple version rows in sequence and confirms each dialog open uses the newly clicked scope.
