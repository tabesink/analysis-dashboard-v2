# DMD-20-01 — Reuse the Edit Metadata panel outside the route

Type: AFK

User stories covered: 2, 3, 4

## What to build

Extract the existing Edit Metadata tab behavior into a reusable scope-driven panel that can be embedded by both the full-page Edit Metadata route and a future modal dialog. The full-page route should continue to behave as it does today, including selecting a program/version from the side panel, loading dynamic metadata options, showing mixed-value placeholders, editing phase and weight fields, copying/resetting values, saving dirty changes, and enforcing admin/write permissions.

The slice is complete when the current route is backed by the reusable panel and there is no user-visible behavior regression.

## Acceptance criteria

- [x] The full-page Edit Metadata route still loads and edits metadata for a selected program/version.
- [x] The reusable panel accepts an externally supplied program/version scope without depending on the route side panel.
- [x] The current route can still own its side-panel selection state and pass the selected scope into the panel.
- [x] Existing save, copy, paste, reset, restore, dirty-state, and toast behavior is preserved.
- [x] Admin-only metadata fields remain admin-only, and users without write access cannot save.
- [x] Existing metadata save cache invalidation still runs after a successful save.
- [x] Focused tests cover the extracted panel or route wrapper enough to catch a regression in scope initialization and save enablement.

## Blocked by

None - can start immediately
