# DMD-22-02 — Enforce channel-map-only upload guardrails

Type: AFK

Blocked by: DMD-22-01

User stories covered: 3, 4, 5

## What to build

Harden the Assign Channels upload popup so it stays a channel-map-only correction path. The popup and backend must reject any upload that is not exactly one file whose basename is `channel_map.yml` or `channel_map.yaml`, and mutation must remain guarded by the same write and ownership rules as manual channel-map save.

The completed slice should be demoable through failure cases: invalid YAML names are rejected, CSV/RSP files are rejected, multiple files are rejected, read-only users cannot start the operation, and rejected uploads do not modify the existing channel map or process artifacts.

## Acceptance criteria

- [ ] Client-side validation rejects zero files, multiple files, CSV/RSP files, folders, and unrelated YAML filenames before submit.
- [ ] Filename validation uses the selected file basename after directory path stripping and accepts only `channel_map.yml` or `channel_map.yaml`, case-insensitively.
- [ ] Backend validation enforces the same exact basename rule and rejects invalid requests even if the client is bypassed.
- [ ] The scoped upload endpoint uses the same write-user and owner/admin authorization rules as manual channel-map save.
- [ ] Read-only users cannot upload channel maps from the Assign Channels panel.
- [ ] Invalid uploads leave the popup open with clear error feedback.
- [ ] Invalid uploads do not mutate `dim_channel_map`, active channel-map snapshots, event data, or retained artifact status.
- [ ] Tests cover frontend validation, backend filename validation, permission denial, and no-mutation behavior on rejection.
