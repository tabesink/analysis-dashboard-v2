# DMD-22-03 — Prove upload parity, refresh, and cache behavior

Type: AFK

Blocked by: DMD-22-01, DMD-22-02

User stories covered: 2, 5

## What to build

Close the feature by proving that the Assign Channels upload path behaves like the existing Upload Data channel-map path where it matters: equivalent YAML maps produce equivalent normalized mappings, pending artifacts become plottable, visible Database indicators refresh, and existing manual Reset/Save behavior remains unchanged.

The completed slice should be verifiable by uploading a channel map in the dialog, observing the missing-channel-map warning clear after processing succeeds, and confirming both the Database table and Dashboard-dependent data refresh without a manual browser reload.

## Acceptance criteria

- [ ] Add regression coverage showing that a YAML map applied through the scoped Assign Channels upload path normalizes to the same plot mappings as the original Upload Data YAML path for the same map definition.
- [ ] Successful scoped upload invalidates or refetches the same dependent client queries as manual channel-map save, including channel-map editor data, Database row indicators, and dashboard/event dependencies.
- [ ] The missing-channel-map warning clears after successful upload and retained-artifact processing when no failures remain.
- [ ] Manual Assign Channels **Reset** and **Save** behavior still works after the Upload button and popup are added.
- [ ] Success feedback distinguishes at least the important outcomes: map applied with processed files, map applied with failed files, and map applied when there were no pending files to process.
- [ ] Failed processing returns clear feedback without closing the popup prematurely or hiding the failed count.
- [ ] Tests cover query invalidation/refetch behavior and UI state after successful and partially failed processing.
