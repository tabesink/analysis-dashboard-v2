# UP-24-01 — Add lean async channel reprocess tasks

Type: AFK

Blocked by: None

User stories covered: 1, 2, 3, 4, 5, 6, 8, 10, 11, 12, 13, 14, 15, 41, 42, 44, 45

## What to build

Convert Assign Channels save and channel-map YAML upload from long synchronous reprocess requests into a lightweight async `channel_reprocess` task backed by the existing upload-task storage pattern.

This slice should keep the app lean: extend the current task row shape with only the fields needed for derived-data tasks, reuse existing polling/ownership ideas, and keep the current per-artifact channel reprocess semantics. The task reads canonical raw load histories, generates cross-plot LTTB data, updates channel-map lineage, and reports coarse progress. It must not calculate damage.

Starting a channel reprocess for a program/version with an already active derived-data task should return the existing active task id instead of launching duplicate work.

## Acceptance criteria

- [ ] Channel-map Save starts a `channel_reprocess` task and returns `{ task_id, task_kind, reused_existing_task }`.
- [ ] Channel-map YAML upload starts the same `channel_reprocess` task path and returns the same task-start shape.
- [ ] `task_kind` is `channel_reprocess` for this path.
- [ ] The task row stores only lean progress state: status, phase, sub-phase, completed events, total events, current event, progress message, scope, and result/error.
- [ ] The task reuses existing retained-artifact reprocessing behavior to write raw load histories, cross-plot LTTB data, and derived-data lineage.
- [ ] The task preserves per-artifact transaction behavior: a failed artifact does not roll back already completed sibling artifacts.
- [ ] The task never calculates load-history damage.
- [ ] If another derived-data task is active for the same program/version, the start response returns that existing task id with `reused_existing_task: true`.
- [ ] Polling is creator-scoped and does not expose another user's task.
- [ ] Poll responses expose only the lean task status payload: task id, task kind, status, phase, sub-phase, progress message, completed events, total events, current event, error, and result.
- [ ] Progress messages use the locked copy where applicable: `Validating artifact 3/10: event_042.csv` and `Generating cross-plot data: event_042 - bj_xy_force_plot (4,872 points)`.
- [ ] Server tests cover task start, active-task reuse, polling ownership, progress updates, partial artifact failure, and unchanged LTTB output compared with the previous synchronous path.
