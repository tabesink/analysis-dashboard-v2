# UP-24-04 — Add schedule damage progress and repair-report UX

Type: AFK

Blocked by: UP-24-03

User stories covered: 7, 8, 9, 10, 11, 16, 17, 18, 19, 20, 23, 24, 25, 36, 37, 38, 39, 40, 44, 45

## What to build

Wire the Durability Schedule UI to the schedule-triggered damage contract from UP-24-03.

Schedule upload/replacement and schedule-row save should show damage progress when a `damage_task_id` is returned. If the server returns a prerequisite report instead, the UI should show that report without polling. If the task fails with a damage validation report, the modal summary should provide a direct action back to the Durability Schedule editor, where affected fields are highlighted and a compact report summary appears above the table. Saving corrected schedule rows automatically starts a fresh damage calculation.

## Acceptance criteria

- [ ] Schedule upload/replacement starts damage progress polling when the response includes `damage_task_id`.
- [ ] Schedule-row save starts damage progress polling when the response includes `damage_task_id`.
- [ ] The damage progress modal uses the locked live message copy: `Calculating load history damage: event_042 - BJ X Force`.
- [ ] If the schedule response includes `damage_prerequisite_report`, the UI shows the report and does not poll for a task.
- [ ] Closing the damage progress modal only hides the modal; processing continues.
- [ ] Active damage work appears in the scoped Edit Metadata inline banner and can reopen progress.
- [ ] A task failure with a damage validation report shows a modal summary with a direct action to reopen the Durability Schedule editor.
- [ ] The Durability Schedule editor shows the report summary above the schedule table.
- [ ] Affected editable fields are highlighted inline using the report's field names.
- [ ] Report display stays compact: event name plus field issue, not row-level current values or suggested replacements.
- [ ] Saving corrected schedule edits automatically starts a new damage calculation.
- [ ] If corrected edits still fail validation, the UI replaces the old report with the updated server report.
- [ ] Read-only users can see damage state/report context but cannot edit schedule rows or retry by saving.
- [ ] Client tests cover schedule upload task start, schedule save task start, prerequisite report display, modal failure summary, editor reopening, field highlights, automatic retry, and updated failure reports.
