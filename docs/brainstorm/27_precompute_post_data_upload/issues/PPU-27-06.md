# PPU-27-06 — Harden precompute idempotency and document rollout

Type: AFK

Blocked by: PPU-27-01, PPU-27-02, PPU-27-03, PPU-27-04, PPU-27-05

User stories covered: 10, 11, 17, 18, 23, 24, 27, 28, 29, 30, 33, 34, 37, 39, 40

## What to build

Close the post-upload precompute loop with end-to-end hardening across the automatic trigger points. Repeated upload completion, channel completion, schedule save, and Inspect Damage access should converge to the same current derived state without duplicate active tasks, synchronous read-time calculation, or stale values being mistaken for current results.

This slice should also document the rollout boundary for future coding agents: the feature remains scoped to program/version derived data, reuses the existing task storage and public task kinds, and does not introduce a generic background processing platform.

## Acceptance criteria

- [ ] Only one active derived-data task per program/version remains enforced across all automatic trigger points.
- [ ] Repeated completion events or repeated page visits do not start duplicate damage tasks.
- [ ] Automatic failures preserve previous successful damage rows as stale when appropriate.
- [ ] Missing or invalid schedule prerequisites never produce partial scheduled damage rows.
- [ ] Unscheduled uploaded events do not trigger unnecessary damage calculation.
- [ ] Stale base damage is never used for rescale-only scheduled damage updates.
- [ ] End-to-end tests cover the main workflow-order permutations: schedule before channel assignment, channel assignment before schedule, Inspect Damage repair, and read-only Inspect Damage no-op.
- [ ] Documentation captures the shipped boundaries, follow-up exclusions, test coverage, and any durable implementation decisions.
- [ ] `CHANGELOG.md`, `docs/master-build-plan.md`, and `docs/tasks/PPU-27-*.md` are updated as required when implementation ships.
