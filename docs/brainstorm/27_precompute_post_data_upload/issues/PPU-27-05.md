# PPU-27-05 — Refresh derived views and surface automatic precompute feedback

Type: AFK

Blocked by: PPU-27-01, PPU-27-02, PPU-27-04

User stories covered: 9, 19, 20, 21, 22, 23, 27, 32, 36, 37, 40

## What to build

Centralize client-side derived-task completion handling so automatic damage work refreshes the views that depend on persisted damage and schedule state. When automatic damage calculation completes, Inspect Damage queries should refresh so newly persisted values appear without a manual page reload, and Edit Metadata schedule-context queries should refresh so schedule-derived state stays current.

Blocked automatic precompute and automatic failures should use concise toast-only feedback for this PRD. Do not add a global task center, heavy blocked-state banner, or new cross-route task UI.

## Acceptance criteria

- [x] Completion of a `damage_calculation` task invalidates Inspect Damage queries for the affected program/version.
- [x] Completion of a `damage_calculation` task invalidates schedule-context queries used by Edit Metadata.
- [x] Completion handling is shared across manual and automatic derived-data task starts where practical.
- [x] Closing the derived-data progress modal keeps background processing running.
- [x] Blocked automatic precompute emits a concise toast that identifies the program/version scope when useful and names the missing prerequisite at a high level.
- [x] Automatic calculation failure uses toast-only visibility unless an existing derived-data flow already provides richer feedback.
- [x] No global background task center, new queue UI, or cross-route persistent task list is added.
- [x] Client tests cover Inspect Damage invalidation, schedule-context invalidation, blocked toast feedback, modal close behavior, and unchanged derived-data modal behavior.
