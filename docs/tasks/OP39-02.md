# OP39-02 — Authorize Derived Task Reuse And Polling By Scope

## Status

DONE (2026-06-17)

## Behavior Changed

- Derived-task polling is now authorized by scope ownership instead of creator id:
  - admins can poll any active `channel_reprocess` or `damage_calculation` task
  - non-admin users can poll derived tasks only when they own/uploaded that program/version scope
- Derived-task start/reuse for damage routes now uses the same scope ownership rule:
  - `POST /api/v1/damage/backfill`
  - `POST /api/v1/damage/calculate`
- Unauthorized users no longer receive reused active damage task ids for scopes they do not own.
- Folder-upload task polling remains creator-scoped and unchanged.

## Interfaces Changed

### Server

- Added `can_access_derived_task_scope(...)` in `server/services/dashboard_orchestration.py` as the shared helper for admin-or-scope-owner task authorization.
- Updated `GET /api/v1/dashboard/derived-data/task/{task_id}` in `server/routers/dashboard.py`:
  - fetch task by id
  - enforce derived-task kind
  - enforce shared scope-ownership authorization
  - return `404` for unauthorized access to avoid leaking task details
- Updated `server/routers/damage.py` to enforce scope ownership before calling backfill/calculate orchestration.

## Tests Added/Updated

- `tests/server/routers/test_dashboard_router.py`
  - renamed unrelated-user poll test for clarity
  - added admin polling regression for non-creator derived task
  - added scope-owner polling regression where creator is a different user
- `tests/server/routers/test_damage_router.py`
  - added unauthorized writer regression proving active-task reuse id is not leaked
  - added admin-start regression for other owner scope
