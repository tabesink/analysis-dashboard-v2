## CU40-06 - Auth Database Responsiveness And Final Regression

### Behavior added

- Login audit persistence is now best-effort and nonblocking for the dashboard DB lock path.
- `POST /api/v1/auth/login` now remains successful when dashboard audit writes are unavailable or contended.
- Database switch/delete admission regression coverage now explicitly includes active `cancelling` upload tasks and stale `cancelling` heartbeat reconciliation.

### Interfaces changed

- Server storage: `UnifiedStore.try_log_audit(...) -> bool` for bounded best-effort audit insertion.
- Server auth route: `POST /api/v1/auth/login` now uses `try_log_audit(..., lock_timeout_seconds=0.05)` and logs a structured warning when audit is skipped.

### Tests added/updated

- `tests/server/routers/test_auth_routes.py`
  - login succeeds when dashboard audit write is unavailable.
  - login remains responsive under dashboard DB lock contention.
- `tests/server/routers/test_export_router.py`
  - database switch is blocked by active `cancelling` folder uploads and includes cancel timing in blocker payloads.
  - database delete reconciles stale `cancelling` rows to failed before admission.

### Residual risk

- Best-effort login audit writes can be intentionally skipped under DB contention windows. This preserves authentication responsiveness and avoids lock-coupled login failures, but operators should rely on structured app logs (`event=login_audit_skipped`) during contention periods.
