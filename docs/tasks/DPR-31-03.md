# DPR-31-03 — Persist simplified damage and task states

Date: 2026-06-12
Status: DONE

## Behavior added or changed

- Schedule-driven damage worker now persists missing-channel cells with `status="unavailable"` instead of collapsing them into generic `error`.
- A single scope run can now persist mixed cell outcomes (`current`, `error`, `unavailable`) while still completing the task when at least one event path remains processable.
- Unexpected runtime exceptions in the worker now persist a structured `failure_report` in `upload_tasks.result_json`, keeping failed task context queryable through existing task/failure readers.

## Interfaces changed

- `DamageCalculationTaskService._run_damage_calculation_task`:
  - unavailable channel-source branch writes `event_channel_damage.status="unavailable"`;
  - unexpected exception branch writes `result={"failure_report": ...}` when marking task `failed`.
- No API schema changes were required for this slice; existing failure-report readers continue to use `result_json.failure_report`.

## Tests added and what they prove

- `test_damage_task_persists_mixed_cell_outcomes_for_one_scope_run`
  - proves one run can persist `unavailable`, `error`, and `current` cells together and still end task state as `completed`.
- `test_damage_task_unexpected_exception_persists_failure_report`
  - proves unexpected worker exceptions end task state as `failed` and persist structured failure context in `result_json.failure_report`.
- Full `tests/server/services/test_damage_calculation_task.py` passes with these additions.

## Follow-on assumptions for next slice

- Inspect read-model work (`DPR-31-04`) can treat persisted cell states as canonical simplified vocabulary (`current|error|unavailable`) for normal rendering flows.
- Failed-task query paths can rely on `result_json.failure_report` for both validation failures and unexpected worker exceptions.

## Decisions intentionally unchanged

- Existing schedule command outcome contract (`schedule_command_outcome`, `damage_task_status`, `damage_prerequisite_report`) remains unchanged.
- Inspect page mutation/backfill behavior is intentionally untouched in this slice.
