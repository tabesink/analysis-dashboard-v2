# DPR-31-01 — Schedule save command starts damage lifecycle

## Behavior added or changed

- Schedule attach/save command responses now expose an explicit lifecycle outcome contract: `schedule_command_outcome`, `damage_task_id`, `damage_task_status`, and `damage_prerequisite_report`.
- Command outcomes are explicit at the backend boundary:
  - `calculation_started` when a new damage task is started.
  - `reused_active_task` when an active task is reused.
  - `validation_blocked` when prerequisites fail and no task starts.
  - `failed_to_start` as a fallback outcome.
- Frontend schedule response resolution now reads explicit command outcomes first, with legacy fallback to existing `damage_task_id` and `damage_prerequisite_report` fields for compatibility.

## Interfaces changed

- Backend model contract:
  - `server/models/dashboard.py`
  - `DurabilityScheduleAttachResponse`
  - `DurabilityScheduleContextResponse`
- Backend decision-extension mapping:
  - `server/services/post_upload_precompute.py`
  - `schedule_precompute_decision_to_extension()`
- Router contract passthrough:
  - `server/routers/dashboard.py`
  - `_schedule_damage_extension()`
- Frontend contract and resolution:
  - `client/src/types/api.ts`
  - `client/src/features/edit-metadata/lib/schedule-damage-response.ts`

- `client/src/features/edit-metadata/lib/schedule-damage-response.test.ts`
