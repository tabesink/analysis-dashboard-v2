# DMD-22-02 — Channel-map-only upload guardrails

**Status:** DONE (2026-06-10)

## Scope

Harden the Assign Channels upload path so it only accepts one channel-map file with basename `channel_map.yml` or `channel_map.yaml`, while preserving write/ownership enforcement and preventing side effects on rejected uploads.

## Implementation

### Frontend

- Added `validateChannelMapUploadSelection(files)` in `channel-map-file.ts` to centralize selection guardrails:
  - reject zero files;
  - reject multiple files;
  - reject folder selections (`webkitRelativePath`);
  - accept only `channel_map.yml` / `channel_map.yaml` basenames after path stripping, case-insensitive.
- Updated `ChannelMapUploadDialog` to use the shared validator and keep inline errors visible in the popup when validation fails.

### Backend

- Updated `POST /api/v1/dashboard/program-version/channel-map/upload` to bind `channel_map` as `list[UploadFile]`.
- Enforced backend cardinality authority with `len(channel_map) != 1` rejection before any processing.
- Kept the existing basename guard (`is_valid_channel_map_filename`) and ownership/write checks.

## Tests

- Added frontend unit coverage:
  - `client/src/features/edit-metadata/lib/channel-map-file.test.ts`
  - verifies basename normalization, invalid YAML rejection, zero/multi-file rejection, folder rejection, and valid file acceptance.
- Added backend router coverage:
  - `test_upload_program_version_channel_map_rejects_multiple_files`
  - `test_upload_program_version_channel_map_forbids_read_only_user`
  - `test_upload_program_version_channel_map_invalid_filename_is_noop`
- Existing happy-path route test remains and still passes.
