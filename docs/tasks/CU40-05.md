## CU40-05 - Reconnect Active Task Discovery And UI Recovery

### Behavior added

- Added `GET /api/v1/upload/tasks/active` to discover recoverable upload-task operations after browser reload/reconnect.
- Discovery now runs stale-heartbeat reconciliation before building the response and returns:
  - `active_tasks` (`queued` / `running` / `cancelling`)
  - `recent_terminal_tasks` (failed/cancelled folder uploads for cleanup/retry reconnect guidance)
- Folder-upload discovery visibility is creator-scoped.
- Derived-task discovery visibility is admin-or-scope-owner scoped.
- Database page startup now fetches discovery and:
  - surfaces a reconnect banner for recoverable folder uploads with **View progress**
  - restores derived task polling state for discovered active scopes.
- Auth redirect and background data-sync polling no longer rely on `folderUploadInProgress` as durable recovery state.

### Interfaces changed

- Server model: `UploadTaskDiscoveryResponse` in `server/models/upload.py`.
- Server storage: `UnifiedStore.list_upload_tasks_for_discovery(...)`.
- Server route: `GET /api/v1/upload/tasks/active` in `server/routers/upload.py`.
- Client API: `uploadApi.getActiveUploadTasks()`.
- Client types: `UploadTaskDiscoveryResponse` and expanded `UploadTaskEvent` lifecycle fields.
- Client hooks/stores:
  - `useUploadOperation` now supports `recoverTask(taskId)`.
  - `useUpload` now supports `recover(taskId)`.
  - derived stores now treat `cancelling` as active and `cancelled` as terminal.
