# P11-10: Startup schema mutation ownership cleanup

## Summary

Clarified startup mutation ownership behind one explicit storage entry point while
preserving runtime behavior, call order, and connection/locking model.

## Changes

- Added `MigrationRunner.initialize_store_for_startup()` in
  `server/storage/migrations.py` as the canonical startup path:
  1. `migrate_up()` for declared schema mutations
  2. construct `UnifiedStore` as the single DuckDB connection owner
  3. apply runtime startup backfills
- Updated `server/main.py` lifespan startup to use
  `initialize_store_for_startup()` instead of manually orchestrating migration +
  store construction in separate steps.
- Updated `server/storage/database.py`:
  - `UnifiedStore.__init__` now supports `initialize_schema` (default `True`)
    to preserve existing call sites.
  - Added `UnifiedStore.apply_startup_backfills()` to allow startup path reuse
    without re-running declared DDL.
- Added startup-path behavior coverage in
  `tests/server/storage/test_schema_initialization.py`:
  `test_migration_runner_startup_path_applies_backfills_without_reapplying_schema`.
