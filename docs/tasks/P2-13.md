# P2-13: Pending Channel-Map Uploads

Status: DONE (2026-04-29)

## Summary

Uploads no longer require `channel_map.yaml` to be present at upload time. When the map is missing, CSV files and RSP-converted CSV bytes are retained under the managed artifact directory and indexed in `ingestion_artifacts` with owner, preview, metadata, and processing status.

The Edit Metadata page now exposes a Channel Map tab with the fixed eight plot definitions. Saving the map validates zero-based column indexes against the retained CSV preview, writes `dim_channel_map`, and processes retained artifacts into normal `dim_event`, `measurements_raw`, and `measurements_lttb` rows.

## Implementation Notes

- `ingestion_artifacts` stores artifact metadata and relative filesystem paths. Artifacts are retained indefinitely for future map edits and reprocessing.
- `dim_channel_map` now stores `x_col` and `y_col` alongside channel labels so manually edited maps can be reapplied deterministically.
- Database list summaries include `missing_channel_map`, pending counts, and failed counts so the Database tree can show warning rollups.
- Dashboard event catalog responses include non-selectable pending placeholders so unmapped program/versions are visible but cannot be plotted.
- Retained artifacts remain source-local. Load-data export/import no longer includes `managed_artifacts/channel-map` or `ingestion_artifacts`, so pending uploads must be processed before export or re-uploaded on the target.

## Verification

- Added ingestion service coverage for missing-map artifact retention.
- Added ingestion service coverage for saving a fixed channel map and processing a pending artifact.
