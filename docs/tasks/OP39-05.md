# OP39-05 - Active user presence switch block

## Behavior changed

- Added lightweight authenticated presence heartbeats at `POST /api/v1/auth/presence/heartbeat`.
- Presence now records active user identity (`user_id`, `username`), active database identity, optional active area, and `last_seen_at`.
- Presence is runtime-only, host-local state with short TTL expiry (`60s`), intentionally separate from persisted session lifetime.
- Database switch admission (`POST /api/v1/export/database/connect`) now blocks with `409` when other active users are present on the currently active database.
- Blocked responses include structured active-user usernames via `detail.blocked_by[].usernames` for switch-denial UI messaging.
- Client runtime now sends periodic best-effort heartbeats while authenticated and app-visible.
- Database switch UI now shows a toast naming active users when switch is blocked by active presence.

## Interfaces changed

- New server module: `server/services/active_presence.py`.
- New auth models:
  - `PresenceHeartbeatRequest`
  - `ActivePresenceResponse`
  - `PresenceHeartbeatResponse`
- New auth route: `POST /api/v1/auth/presence/heartbeat`.
- Operation-admission blocker payload now supports optional `usernames`.
- `assert_can_switch_or_delete_database(...)` now accepts optional `requesting_user_id`.
- Client auth API adds `authApi.heartbeatPresence(activeArea?)`.

## Residual risks / follow-ups

- Presence is intentionally in-process memory only; multi-instance deployment would require shared presence state.
- Presence heartbeat is best-effort and visibility-gated client-side; transient network failures can delay stale-user clearance until TTL expiry.
- Same-user multi-tab/session differentiation is not implemented; presence is currently keyed by `user_id`.
