# DPR-31-04 — Inspect API is strict read model

## Behavior changed

- `POST /api/v1/damage/inspect` remains a query-only path over persisted `event_channel_damage` rows and selected event metadata.
- Inspect scope hydration no longer invokes repair/prerequisite policy helpers during reads.
- Inspect response still exposes read-only running/failed task context through `active_damage_task_id` and `failure_report`.

## Interfaces changed

- Internal service behavior in `server/services/damage_inspect.py` changed: `_build_scope_state()` no longer calls:
  - `assess_scope_damage_repair_state()`
  - `check_damage_prerequisites()`
- Public response shape remains compatible for existing scope fields in this slice.

## Tests added

- `test_damage_inspect_is_query_only_and_skips_repair_prerequisite_checks`
  - Proves inspect read path does not call repair/prerequisite checks and still returns selected rows with empty damages.
- `test_damage_inspect_surfaces_running_and_failed_task_context`
  - Proves inspect scope metadata exposes running task id and latest failed-task report as read-only context.

## Follow-on assumptions for next slice

- Frontend inspect flow can rely on inspect API as non-mutating and remove page-load backfill/calculate triggers.
- Running/failed banners can use `active_damage_task_id` and `failure_report` from inspect scope metadata.

## Decisions intentionally unchanged

- Schedule upload/save remains the command boundary that starts or reuses calculation tasks.
- Damage worker persistence semantics (`current`, `error`, `unavailable`) remain unchanged from DPR-31-03.
