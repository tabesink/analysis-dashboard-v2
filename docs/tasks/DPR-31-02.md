# DPR-31-02 — Clear stale scope damage and dedupe active tasks

## Behavior added or changed

- Schedule save precompute now clears persisted `event_channel_damage` rows for the accepted `program_id/version` scope before starting or reusing a `damage_calculation` task.
- Command boundary behavior preserves active-task dedupe (`reused_active_task`) while ensuring stale scope rows are removed ahead of the next calculation lifecycle.
- Failed damage-task validation after schedule save no longer leaves prior persisted damage rows behind from older schedules.

## Interfaces changed

- Backend orchestration:
  - `server/services/post_upload_precompute.py`
  - `decide_after_schedule_save()`
- Unified store scope cleanup helper:
  - `server/storage/database.py`
  - `clear_event_channel_damage_for_program_version()`

## Tests added and what they prove

- `tests/server/routers/test_durability_schedule_router.py`
  - `test_put_schedule_save_clears_scope_damage_and_reuses_active_task` proves the public schedule-save command path clears scope damage rows and reuses the active task without creating a second running task.
- `tests/server/services/test_post_upload_precompute.py`
  - `test_decide_after_schedule_save_invalid_rows_fail_validation_without_partial_results` updated expectation verifies invalid schedule rows fail task validation without retaining old persisted damage rows after scope cleanup.

## Follow-on assumptions for next slice

- `DPR-31-03` can assume accepted schedule command paths start from a clean persisted damage scope when entering calculation execution.
- One-active-task command dedupe semantics remain enforced at the schedule command boundary.

## Decisions intentionally left unchanged

- Inspect Damage read response semantics remain unchanged in this slice.
- Frontend Inspect Damage behavior remains unchanged in this slice.
- Fatigue damage formula/channel math semantics remain unchanged.
