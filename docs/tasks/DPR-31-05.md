# DPR-31-05 — Inspect Damage UI uses read-only lifecycle states

## Behavior added or changed

- Removed automatic Inspect Damage page-load backfill behavior so opening `/inspect-damage` no longer triggers hidden damage mutation work.
- Updated Inspect Damage status presentation to show explicit running lifecycle context from read-only inspect scope metadata (`active_damage_task_id`).
- Updated Inspect Damage table cell rendering so persisted `unavailable` cells render clearly instead of appearing blank.
- Updated loading copy to clarify read behavior (`Loading saved damage results`) while calculations are running.

## Interfaces changed

- `resolveInspectDamageViewState()` now exposes `runningScopes` derived from inspect scope task metadata.
- `planInspectDamageBackfillAttempts()` now returns no scopes (read-only lifecycle policy on inspect load).
- `isDamageCellDisplayable()` now treats `unavailable` as renderable.

## Tests added and what they prove

- `client/src/features/inspect-damage/__tests__/inspect-damage-view-state.test.ts`
  - validates running scope metadata is surfaced via view state.
  - validates `unavailable` damage cells are displayable.
- `client/src/features/inspect-damage/__tests__/plan-inspect-damage-backfill-attempts.test.ts`
  - validates inspect load planning produces no automatic mutation attempts.

## Follow-on assumptions for next issue

- Inspect Damage remains a strict read surface for lifecycle state; mutations are schedule/upload command owned.
- Running and failed task metadata continue to come from inspect read responses and are suitable for status UX hardening.

## Decisions intentionally left unchanged

- Manual calculate action wiring and damage-task modal flow remain unchanged.
- Backend calculation orchestration and fatigue math remain unchanged.
