# P8-07: Optimistic concurrency control on updates

## Summary

Implemented optimistic concurrency control for single-event metadata updates to
prevent lost updates in multi-user flows.

## Changes

- Added OCC request contract for single-event metadata updates:
  - `EventMetadataUpdateRequest` now requires `if_unmodified_since`
  - kept program-version bulk update payload shape unchanged by splitting shared
    metadata fields into `EventMetadataUpdateFields`
- Added storage-level atomic compare-and-update:
  - `UnifiedStore.update_event_if_unmodified(...)`
  - updates succeed only when current `updated_at` matches caller token
  - supports `NULL` tokens for never-updated rows
- Added service-layer conflict handling:
  - `QueryService.update_event_metadata(...)` now accepts
    `if_unmodified_since`
  - raises `OptimisticConcurrencyError` when token is stale
- Added route-level conflict mapping:
  - `PUT /api/v1/dashboard/events/{event_id}/metadata` now returns
    `409 Conflict` on optimistic concurrency failures
- Added regression coverage:
  - `test_update_event_metadata_rejects_stale_if_unmodified_since`
  - `test_update_event_metadata_returns_409_on_optimistic_concurrency_conflict`
