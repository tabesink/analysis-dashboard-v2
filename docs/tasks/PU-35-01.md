# PU-35-01 — Enforce one Reference/Target program-version scope

## Behavior added or changed

- Inspect Damage Reference/Target sidepanel selection now enforces a one-scope invariant per side (`program_id` + `version`).
- Selecting multiple events within the same scope still works normally.
- Selecting an event from a different scope replaces that side with the new scope instead of silently mixing scopes.
- Clearing a side (`[]`) still resets the active scope for that side.
- When cross-scope replacement occurs, the UI now shows a toast explaining that only one scope per side is allowed and the selection moved to the new scope.

## Interfaces changed

- Updated `client/src/components/dashboard/side-panel/ComparisonLoadDataSections.tsx`:
  - Added `enforceSingleProgramVersionScope()` helper.
  - Added `selectedIdsEqual()` helper for replacement detection.
  - Updated `updateSelection()` to normalize mixed-scope selections and emit explicit feedback.

## Tests added and what they prove

- Expanded `client/src/components/dashboard/side-panel/ComparisonLoadDataSections.test.tsx` with new behavior tests:
  - same-scope multi-select remains selectable (`event-1`, `event-2` in `P1/V1`)
  - cross-scope selection replaces Reference (`event-1` + `event-3` -> `event-3`)
  - clearing Reference emits empty selection (`[]`)
  - cross-scope selection replaces Target (`event-1` + `event-3` -> `event-3`)

## Follow-on assumptions for next slice

- `comparison.reference.selected_event_ids` and `comparison.target.selected_event_ids` are each constrained to at most one program/version scope.
- `PU-35-02` can build Plot Inputs behavior on top of this invariant without adding a version selector.

## Decisions intentionally left unchanged

- No changes to Global Filters behavior.
- No changes to Dashboard route Load Data behavior.
- No changes to damage calculation APIs or plot rendering.
