# OP39-03 — Failed Upload Cleanup And Safe Retry

## Status

DONE (2026-06-17)

## Behavior Changed

- Failed `folder_upload` task polling now includes explicit cleanup/retry guidance in `error_details`:
  - task reports whether partial committed events were detected
  - task reports how many candidate events can be cleaned
  - task provides a cleanup endpoint path and retry guidance message
- Startup-reconciled failed folder uploads (for example `Task interrupted by server restart`) now expose the same retry guidance contract even when no committed events are recorded yet.
- New cleanup route allows safe retry preparation without deleting unrelated successful data:
  - `POST /api/v1/upload/folder/task/{task_id}/cleanup`
  - owner of the failed upload or an admin can run cleanup
  - unrelated users are rejected
- Folder upload task progress now records partial committed event ids while ingestion is running, so failed/interrupted tasks retain cleanup candidates.

## Interfaces Changed

### Server

- `UploadTaskEvent.error_details` now includes failed-upload cleanup/retry keys when `task_kind=folder_upload` and `status=failed`.
- Added `FailedUploadCleanupResponse` (`server/models/upload.py`).
- Added `UnifiedStore.cleanup_failed_folder_upload_task(...)` to hard-delete only event/artifact rows tied to the failed task's recorded committed events.
- `IngestionService.start_upload_task(...)` now stores upload scope (`program_id`, `version`) and progressive partial `event_ids` in `upload_tasks.result_json` while events are committed.
- Added `POST /api/v1/upload/folder/task/{task_id}/cleanup` in `server/routers/upload.py`.

### Client/API

- Added `uploadApi.cleanupFailedFolderUploadTask(taskId)` helper.
- `uploadApi.waitForUploadTask(...)` now appends server-provided retry guidance to failed upload error messages.
- Added `FailedUploadCleanupResponse` and typed `UploadTaskErrorDetails` in `client/src/types/upload.ts`.

## Residual Risks

- Cleanup candidate detection relies on event ids recorded in task `result_json`; legacy failed tasks created before this slice may have no candidate ids and therefore return zero deletions.
- Cleanup currently removes files recorded in `ingestion_artifacts.artifact_path`; orphaned source/canonical files not linked through failed-task event ids remain outside this scoped cleanup path.
