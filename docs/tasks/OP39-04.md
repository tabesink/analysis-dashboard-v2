# OP39-04 - Operation admission guards

## Behavior changed

- Added a shared server-side operation admission guard layer in `server/services/operation_admission.py`.
- `POST /api/v1/export/database/connect` and `POST /api/v1/export/database/delete` now reject with `409` when active folder uploads, derived tasks, database exports, or another exclusive database operation are running.
- `POST /api/v1/export/database/create-new` now runs under exclusive admission and rejects with `409` when another exclusive database operation is in progress.
- `POST /api/v1/export/database/parquet/export/start` now rejects with `409` while an exclusive database operation is running.
- `POST /api/v1/upload/folder/start` now rejects with `409` while an exclusive database operation is running.
- Derived-task starts (channel reprocess and damage calculation) now reject with `409` while an exclusive database operation is running.
- Blocked responses now return structured `detail` payloads for toast-friendly client rendering:
  - `code`
  - `operation`
  - `message`
  - `blocked_by[]`

## Interfaces changed

- New service module: `server/services/operation_admission.py`.
- New export-service helper: `list_tasks(kinds=None, statuses=None)` in `server/services/export.py`.
- Added shared task-kind constants for exclusive database operations in `server/upload/task_kinds.py`:
  - `database_create`
  - `database_switch`
  - `database_delete`

## Residual risks / follow-ups

- Exclusive create/switch/delete guards are in-process only and intentionally non-durable for this phase.
- Active-user presence blocking for database switch is intentionally deferred to `OP39-05`.
- Admission decisions are currently returned via route `HTTPException(detail=...)`; no global typed error envelope abstraction was introduced in this slice.
