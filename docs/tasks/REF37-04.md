# REF37-04 - Folder upload progress and cancellation hardening

## Behavior changed

- Folder-upload progress now follows backend folder phases directly in UI order: `upload_received` -> `converting` -> `validating` -> `writing`.
- Progress updates are monotonic, so out-of-order polling payloads do not regress the visible phase or percentage.
- Non-folder task payloads are no longer rendered as folder-upload progress; the upload dialog keeps waiting on folder-upload state instead of flip-flopping to downstream progress.
- Upload cancellation now uses the existing hook abort contract but lands in an explicit terminal summary in the operation dialog (instead of silently closing).
- Upload toasts are now lightweight lifecycle signals only (start, complete, failed, cancelled) while detailed status remains in the dialog.

## Interfaces changed

- `UploadProgressPhase` now maps to backend folder phase names in `client/src/types/upload.ts`.
- `UploadTaskEvent` now includes optional `task_kind` and supports `cancelled` status to safely handle broader task payloads without displaying unrelated progress.
- `useUpload` now exports `applyUploadTaskProgress(data, previous)` as a pure progress reducer used by the hook and tests.
- `useUploadOperation` now uses dedicated terminal-result builders for failed/cancelled summaries and keeps cancel flow inside the existing modal lifecycle.
- Existing upload polling API contract remains unchanged; cancellation uses the existing abort behavior in the public hook path.

## Tests added

- Added `client/src/hooks/use-upload.test.ts`:
  - verifies backend-phase ordering remains monotonic under out-of-order poll events.
  - verifies downstream task events are not rendered as folder-upload progress.
  - verifies backend `converting` phase is rendered in-order.
- Extended `client/src/features/database-upload/upload-completion-result.test.ts`:
  - verifies terminal failed summary output.
  - verifies terminal cancelled summary output.
- Existing `client/src/lib/api/upload.test.ts` transient polling-retry coverage remains green and continues to assert user-readable connection-lost messaging.

## Follow-on assumptions for next slice

- Folder-upload dialog state is now stable enough to keep upload-lane progress separate from derived-data progress.
- Any future explicit server-side cancel endpoint can be wired without changing the dialog summary contract introduced in this slice.

## Decisions intentionally unchanged

- No server upload route shape changes.
- No channel-map reprocess or damage-calculation semantics changes.
- No generic task-runner refactor or broad module moves.
