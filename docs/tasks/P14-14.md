# P14-14: Host-local identity store for runtime DB switching

Date: 2026-06-10
Status: DONE

Related decision: DEC-084.

## Goal

Keep user accounts and auth sessions stable when operators create or connect to
different managed `dashboard*.db` files.

## Implementation Notes

- Added `identity.db` under `settings.data_root` as the host-local source of
  truth for auth users.
- Rewired `AuthService`, `UserService`, and username display lookup to use the
  identity store instead of the active dashboard store.
- Kept `SessionManager`, saved filters, audit rows, custom fields, and load data
  scoped to each dashboard database.
- Migrated legacy per-dashboard users on startup, preferring the active
  `dashboard.db` on username conflicts.
- Remapped known user-reference columns in managed dashboard databases to the
  selected global identity IDs.
- Removed per-dashboard admin bootstrap from runtime create/connect flows.
- Added narrow user-store protocols so auth/user services and username display
  lookup depend on identity capabilities rather than the concrete dashboard
  store.
- Materialized identity read results under the store lock to avoid shared
  DuckDB connection fetches escaping the identity-store critical section.

## Verification

- Auth/user service tests cover the identity-backed user lifecycle.
- Identity storage tests cover legacy user merge, active-database conflict
  precedence, and reference remapping.
- Export router regression tests cover login persistence after database connect.
