## CU40-03 - Cooperative Folder Upload Cancellation

### Behavior Added
- Folder upload workers now perform cooperative cancel checks at safe checkpoints in `IngestionService.ingest(...)`:
  - before per-file conversion/validation work
  - before event commit writes
  - after each committed event
- Folder upload background task runner now transitions to terminal `cancelled` when an accepted cancel request is observed by the worker, with task result metadata that distinguishes:
  - cancelled before commit (`cleanup_required=false`)
  - cancelled after partial commit (`cleanup_required=true`, candidate count > 0)
- Cancelling progress copy now communicates safe stop semantics while the worker finishes the current write boundary.
- Folder upload task polling now exposes cleanup/retry guidance for cancelled partial uploads (not just failed partial uploads).
- Failed-upload cleanup endpoint now accepts cancelled folder-upload tasks and preserves owner/admin authorization semantics.
- Cleanup data deletion scope remains task-linked via recorded task event ids, preventing unrelated successful data from being removed.

### Client Contract Changes
- `useUpload` cancel behavior now calls server cancel once a task id exists, then keeps polling for server terminal status.
- Browser abort only applies before a server task id exists; task cancellation UI no longer assumes cancellation solely from request abort.
- Upload progress rendering now shows `Cancelling safely...` while task status is `cancelling`.
- Folder-upload polling now treats `cancelled` as terminal (alongside `completed` and `failed`).
- Upload operation modal cancel action switches to non-dismissible cancellation-in-flight copy.
