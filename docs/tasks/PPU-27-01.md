# PPU-27-01 — Auto-start damage after channel reprocess completion

**Status:** DONE (2026-06-11)

## Summary

Introduced a server-side post-upload precompute orchestrator and wired it into channel reprocess task completion so schedule-first workflows can converge to damage calculation once channel-derived prerequisites are current.

## Implementation

- `server/services/post_upload_precompute.py`
  - `decide_after_channel_reprocess_completion()` returns deterministic actions: `no_op`, `blocked`, `start_damage_calculation`, `reuse_active_task`.
  - `run_after_channel_reprocess_completion()` applies the decision using existing `DamageCalculationTaskService`.
- `server/services/ingestion.py`
  - Channel reprocess background task calls `run_after_channel_reprocess_completion()` after marking the task completed.
- `server/services/damage_calculation_task.py`
  - `_start_damage_calculation_task()` accepts optional `reuse_task_kinds`; post-channel-reprocess path reuses only active `damage_calculation` tasks (not in-flight `channel_reprocess` rows).

## Tests

- `tests/server/services/test_post_upload_precompute.py`
  - Decision matrix: no schedule, ready prerequisites, blocked prerequisites, active-task reuse, no-op when damage current.
  - End-to-end: schedule before channel assignment → channel completion auto-starts damage task.
  - Stale preservation when channel reprocess marks prior damage stale.

## Notes

- Public derived task kinds remain `channel_reprocess` and `damage_calculation`.
- Client toast wiring for blocked automatic precompute is deferred to PPU-27-05.
