# P8-15 Implementation Notes

Date: 2026-03-09
Status: DONE

## Scope

- Optimize GVWR/FGAWR/RGAWR range filtering so selected buckets are evaluated in SQL against raw values.
- Avoid per-record application loops when applying range filters.
- Keep existing global filter request shape unchanged.

## Changes Made

### 1) Shared weight-range SQL helper

- Added:
  - `server/utils/weight_filters.py`
- Includes:
  - filter-key to raw-column mapping:
    - `gross_vehicle_weight_range_lbs` -> `gvw`
    - `fgawr_range_lbs` -> `fgawr`
    - `rgawr_range_lbs` -> `rgawr`
  - range-label parser for `low-high` labels
  - SQL predicate builder using `TRY_CAST(NULLIF(TRIM(<raw_column>), '') AS DOUBLE)`

### 2) QueryService events filtering path

- Updated:
  - `server/services/query.py`
- `QueryService.get_events()` now checks weight-range filter keys first and injects SQL predicates against raw value columns.
- Non-weight filters continue to use existing schema/custom-field behavior.

### 3) Program/Version filter paths in DB store

- Updated:
  - `server/storage/database.py`
- `UnifiedStore.get_program_ids()` and `UnifiedStore.get_versions()` now apply the same shared SQL range logic for weight buckets.
- This keeps bidirectional filter behavior consistent across endpoints.
