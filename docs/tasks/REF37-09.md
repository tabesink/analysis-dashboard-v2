# REF37-09 - Task Kind Constants And Startup Reconciliation

Date: 2026-06-16
Status: DONE

## Behavior Added Or Changed

- Added shared upload/derived/database task kind constants in `server/upload/task_kinds.py`.
- Unified active derived task matching and task-kind checks around shared constants, preserving one-active-derived-task-per-program/version semantics.
- Added startup backfill reconciliation so stale persisted active task rows (`queued`/`running`) are marked terminal (`failed`/`failed`) for active task kinds (`folder_upload`, `channel_reprocess`, `damage_calculation`).
- Reconciled startup rows now clear transient progress fields and set a default error message (`Task interrupted by server restart`) when none exists.

## Interfaces Changed

- No endpoint paths or payload shapes changed.
- Internal constant source changed:
  - `server/upload/policies.py` consumes shared task-kind constants.
  - `server/services/derived_data_task.py`, `server/services/damage_calculation_task.py`, `server/services/ingestion.py`, `server/services/post_upload_precompute.py`, `server/routers/dashboard.py`, `server/services/damage_inspect.py`, and `server/storage/database.py` now consume shared constants.

## Tests Added

- `tests/server/unit/routers/test_upload_policies.py`
  - Added shared task-kind contract stability test.
  - Verifies active statuses/kinds and removed `database_import` task-kind constant.
- `tests/server/storage/test_schema_initialization.py`
  - Added startup reconciliation test covering folder + derived active rows becoming terminal on restart.
  - Added startup reconciliation test proving stale derived rows no longer block active derived-task lookup.

## Next Slice Assumptions

- Task-kind names are now centralized and can be reused by orchestration cleanup without string-literal drift.
- Startup no longer leaves stale active task rows that could mislead progress polling or active-task reuse checks.
