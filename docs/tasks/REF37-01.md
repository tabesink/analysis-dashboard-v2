# REF37-01 - Folder upload write permission

## Behavior changed

- Hardened `POST /api/v1/upload/folder/start` to require `WriteUserDep` (write user or admin), not just authenticated user.
- Read-only authenticated users now receive `403` before route payload parsing, validation, or upload task creation begins.
- Write/admin users keep the existing route URL and success response contract (`{ "task_id": "..." }`).

## Interfaces changed

- Server route dependency for `start_upload_folder` in `server/routers/upload.py` now uses `WriteUserDep`.
- Public endpoint URL and response shape are unchanged.

## Tests added

- Added `test_folder_upload_start_forbids_read_only_user_before_parsing_or_task_creation` in `tests/server/routers/test_upload_router.py`.
  - Proves read-only users get `403`.
  - Proves `_parse_upload_payload` does not run for forbidden users.
  - Proves no new upload task rows are created for forbidden users.
- Added `test_folder_upload_start_allows_write_user_with_existing_response_shape`.
  - Proves write users can still start folder upload.
  - Proves the response still returns a non-empty `task_id`.

## Follow-on assumptions for next slice

- Folder upload entrypoint authorization is now a stable invariant (write/admin only) for any upcoming upload policy extraction.
- Later slices can safely refactor upload validation helpers without reintroducing read-only write-path access.

## Decisions intentionally unchanged

- File classification and mixed CSV/RSP rejection logic.
- Ingestion task creation and background execution behavior for authorized users.
- Polling/SSE task status behavior and endpoint contracts.
