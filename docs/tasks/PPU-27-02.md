# PPU-27-02 — Route schedule saves through precompute decisions

**Status:** DONE (2026-06-11)

## Summary

Extended the post-upload precompute orchestrator to schedule upload/replacement and schedule-row save paths so damage calculation starts only after schedule data is persisted and prerequisites are evaluated deterministically.

## Implementation

- `server/services/post_upload_precompute.py`
  - `decide_after_schedule_save()` returns `blocked`, `start_damage_calculation`, or `reuse_active_task` after schedule persistence.
  - `schedule_precompute_decision_to_extension()` maps decisions to API response fields (`damage_task_id`, `damage_prerequisite_report`).
- `server/routers/dashboard.py`
  - `_schedule_damage_extension()` now calls `decide_after_schedule_save()` instead of `maybe_start_after_schedule_change()` for attach and save routes.

## Tests

- `tests/server/services/test_post_upload_precompute.py`
  - Ready prerequisites start damage.
  - Missing channel prerequisites return blocked without creating a task.
  - Active `damage_calculation` task reuse.
  - Schedule-row save and upload persist edits before precompute decision.
  - Invalid schedule rows fail whole-task validation without partial damage results.

## Notes

- Schedule save does not no-op when damage is current (unlike channel-reprocess completion); schedule edits require recalculation.
- Manual Calculate Damage (`damage.py`) still uses `maybe_start_after_schedule_change()` directly.
- Client toast wiring for blocked automatic precompute remains deferred to PPU-27-05.
