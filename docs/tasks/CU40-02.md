## CU40-02 - Cancel Request Policy And Endpoints

### Behavior Added
- Added canonical cancel endpoint `POST /api/v1/upload/tasks/{task_id}/cancel` for `folder_upload`, `channel_reprocess`, and `damage_calculation` task kinds.
- Added compatibility alias routes:
  - `POST /api/v1/upload/folder/task/{task_id}/cancel`
  - `POST /api/v1/dashboard/derived-data/task/{task_id}/cancel`
- Cancel requests are idempotent:
  - `queued`/`running` -> task becomes `cancelling` and `cancel_requested_at` is recorded.
  - `cancelling` -> returns existing `cancelling` state without mutating timestamp.
  - `completed`/`failed`/`cancelled` -> returns current terminal state without mutation.

### Authorization Contract
- Folder upload cancel:
  - allowed for task owner
  - allowed for admin
  - denied for unrelated non-admin users (`403`)
- Derived task cancel (`channel_reprocess`, `damage_calculation`):
  - allowed for admin
  - allowed for program/version scope owner
  - denied for unrelated users with existing hidden-task semantics (`404`)

### Interfaces Changed
- Added server response model: `UploadTaskCancelResponse` in `server/models/upload.py`.
- Added shared policy helper: `has_upload_task_cancel_policy(...)` in `server/upload/policies.py`.
- Added store helper: `UnifiedStore.request_upload_task_cancel(...)` in `server/storage/database.py`.
- Added client types/methods:
  - `UploadTaskCancelResponse` in `client/src/types/upload.ts`
  - `uploadApi.cancelUploadTask(...)`
  - `uploadApi.cancelFolderUploadTask(...)`
  - `derivedDataApi.cancelDerivedDataTask(...)`
- Updated `UploadTaskEvent.status` client typing to include `cancelling`.
