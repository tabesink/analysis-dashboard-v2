# SPV-33-03 — Plotted-channel analysis selection

## Behavior added or changed

- Added `ComparisonPlottedChannelSection` as a dedicated side-panel section for plotted damage channel analysis state.
- The section uses canonical damage channel keys/labels (from `DAMAGE_CHANNELS`) and writes only `comparison.selected_channel_keys`.
- Added explicit guidance in the section clarifying that table metadata/layout columns (for example Work Order and Program ID) do not control plotted-channel meaning.
- Added deterministic pruning support via `pruneSelectedPlottedChannelKeys()` so missing keys are dropped when canonical options change.

## Interfaces changed

- Added `client/src/components/dashboard/side-panel/ComparisonPlottedChannelSection.tsx` with:
  - `CANONICAL_DAMAGE_CHANNEL_OPTIONS`
  - `pruneSelectedPlottedChannelKeys()`
  - `ComparisonPlottedChannelSection` component
- Updated side-panel barrel export to include `ComparisonPlottedChannelSection`.

## Tests added and what they prove

- Added `client/src/components/dashboard/side-panel/ComparisonPlottedChannelSection.test.tsx` covering:
  - canonical plotted-channel option contract and exclusion of metadata column keys
  - channel toggle patching shape for `selected_channel_keys`
  - clear action patching shape without touching unrelated comparison fields
  - missing-key pruning behavior for canonical list changes

## Follow-on assumptions for next slice

- Route wiring can use `ComparisonPlottedChannelSection` as the plotted-channel selector in comparison side panels.
- `onUpdateComparison` remains the route-level merge path so `selected_channel_keys` updates compose with existing `mergeDamageComparisonState()` behavior.

## Decisions intentionally left unchanged

- No changes to generic table column visibility semantics.
- No changes to damage math, aggregation logic, or event-tree selection behavior.
- No backend API changes.
