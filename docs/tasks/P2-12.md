# P2-12: Direct RSP upload conversion through existing CSV ingestion

Date: 2026-04-29
Status: DONE

## Goal

Allow users to upload `.rsp` files directly from the Database upload panel while preserving the existing explicit `channel_map.yaml/.yml` contract for plot semantics.

## Implementation

- Added `RSPConverter`, a focused server-side converter that uses `rpc-reader` to decode uploaded `.rsp` bytes and emits the tagged CSV structure already supported by `CSVParser`.
- Extended the existing `/api/v1/upload/folder/start` upload path to accept either all `.csv` or all `.rsp` data files, ignore unrelated folder contents, and reject mixed CSV/RSP batches.
- Normalized `.rsp` files inside `IngestionService` before parse/validate/write, preserving the original `.rsp` filename as `source_file` and hashing original bytes for duplicate detection.
- Added a simple `converting` upload-task phase so the frontend can show “Converting RSP files...” before validation and event writing.
- Updated the Database upload UI with separate file and folder selectors, CSV/RSP preflight validation, ignored-file messaging, and unchanged metadata/channel-map requirements.
- RSP channel titles come from `rpc-reader` channel `Description` metadata (`DESC.CHAN_N`) before falling back to generic `channel_N` labels.

## Design Decisions

- `.rsp` uploads still require `channel_map.yaml/.yml`; automatic channel-map generation was rejected to avoid silently assigning incorrect plot semantics.
- Raw `.rsp` and converted `.csv` files are not persisted. Conversion is temp-only and feeds the existing ingestion pipeline.
- The existing upload endpoint and SSE task stream were extended rather than adding a separate RSP endpoint.
